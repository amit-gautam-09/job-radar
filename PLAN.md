# job-radar — Build Plan for Claude Code

A daily job-monitoring pipeline that polls the public ATS APIs of ~150 target companies,
filters for GTM-Engineer-adjacent roles that an India-based candidate can actually get
(global-remote, India-eligible, or US roles with visa sponsorship), diffs against
previously-seen postings, and pushes new matches to Telegram at 5:00 PM IST daily via
GitHub Actions. Zero servers. Near-zero cost.

**Owner:** Amit Gautam — B.Tech @ DTU (2027), founder-turned-PM, currently GTM Engineer
Intern @ ClickPost. Goal: remote role at a US-based company; onsite-with-sponsorship is a
bonus, not the strategy.

## Ground rules for Claude Code

These override any instinct to improvise:

1. **No browser automation. No AI-driven page navigation.** This system is a
   deterministic poller against JSON APIs. If an endpoint doesn't exist for a company,
   mark it `unresolved` and move on — do not reach for Playwright or an agent.
2. **Verify field names against live responses before writing parsers.** The endpoint
   shapes below are correct as of mid-2026 but must be confirmed: fetch 2–3 real
   companies per ATS first, inspect the JSON, then write the normalizer. Never code
   parsers from this document alone.
3. **Be polite.** One run per day. Max 10 concurrent requests. 15s timeout, 2 retries
   with exponential backoff. User-Agent: `job-radar/1.0 (personal job alerts)`. These
   are public posting APIs built for exactly this use, but don't hammer them.
4. **Never commit secrets.** Tokens live only in GitHub Actions secrets and local
   `.env` (gitignored). If you see a token in a file about to be committed, stop.
5. **Every filter rule gets a test.** The tricky cases are listed in §Test Cases and
   must pass before Phase 3 is considered done.
6. **Fail loud.** A run that silently misses postings is worse than one that crashes.
   Health problems go to Telegram.

## Data model

Normalized `Job` (dataclass / dict):

```
id: str            # "{ats}:{slug}:{external_id}"  — stable across runs
company: str
title: str
location_raw: str  # exactly as the ATS returns it
workplace: str     # onsite | hybrid | remote | unknown (from ATS fields where present)
url: str           # canonical application URL
posted_at: str     # ISO date if the ATS provides it, else ""
description: str   # plain text, truncated to ~4000 chars (needed for eligibility scan)
source_ats: str
eligibility: str   # set by filters.py — see §Eligibility
title_tier: int    # 1/2/3, or 0 = no match
flags: list[str]   # e.g. ["senior_downrank", "sponsorship_positive"]
score: int|None    # AI layer, 0–100, None if AI disabled
score_reason: str  # one line from AI layer
```

## ATS endpoint reference

All are unauthenticated GETs unless noted. **Confirm shapes live before coding (Ground
Rule 2).**

| ATS | Endpoint | Notes |
|---|---|---|
| Greenhouse | `https://boards-api.greenhouse.io/v1/boards/{slug}/jobs?content=true` | `content=true` embeds the HTML description — required for eligibility scanning. Fields: `jobs[].title`, `location.name`, `absolute_url`, `id`, `updated_at`, `content`. Strip HTML to text. |
| Lever | `https://api.lever.co/v0/postings/{slug}?mode=json` | Array. Fields: `text` (title), `categories.location`, `categories.commitment`, `workplaceType`, `hostedUrl`, `descriptionPlain`, `id`, `createdAt`. |
| Ashby | `https://api.ashbyhq.com/posting-api/job-board/{slug}` | Fields: `jobs[].title`, `location`, `isRemote`, `jobUrl`, `publishedAt`. Check whether description ships inline (`descriptionHtml`) or needs the per-job endpoint; prefer whatever avoids N extra calls. |
| Workable | `https://apply.workable.com/api/v1/widget/accounts/{slug}` | Lists jobs + locations; descriptions may need a per-job fetch — only fetch descriptions for title-tier matches to stay polite. |
| SmartRecruiters | `https://api.smartrecruiters.com/v1/companies/{slug}/postings` | Paginated (`offset`). Descriptions via `/postings/{id}` — again, fetch only for title matches. |
| Recruitee | `https://{slug}.recruitee.com/api/offers/` | Fields under `offers[]`. |
| Workday | *(Phase 5, optional)* `POST https://{tenant}.wd{N}.myworkdayjobs.com/wday/cxs/{tenant}/{site}/jobs` with `{"appliedFacets":{},"limit":20,"offset":0,"searchText":""}` | Covers the enterprise tail (Salesforce, ServiceNow, Adobe, NVIDIA, Workday itself). Fiddly: tenant + site discovery, pagination. Skip in v1. |

**Description policy:** eligibility scanning needs description text, but fetching every
description for every posting is wasteful on ATSes that require per-job calls. Rule:
run the **title filter first**; fetch/scan descriptions only for tier 1–3 title matches.

## Eligibility classifier — the part that matters most

Reality this system encodes: **most "Remote" postings from US companies mean
"Remote (US)"** and require US work authorization. The classifier's job is to separate
the genuinely reachable roles from the mirage.

Enum (evaluated in this order, first confident match wins):

| Value | Signals (regex over `location_raw` + description) |
|---|---|
| `SPONSORSHIP_NEGATIVE_ONSITE` | onsite/hybrid AND negated sponsorship: `unable to (provide )?sponsor`, `no (visa )?sponsorship`, `without (the need for )?sponsorship`, `must be authorized to work in the (US\|United States)( without sponsorship)?` |
| `INDIA_ELIGIBLE` | location contains India \| Bengaluru \| Bangalore \| New Delhi \| Delhi \| Gurgaon \| Gurugram \| Noida \| Mumbai \| Pune \| Hyderabad \| Chennai, or `Remote[ -]*(India\|APAC)` |
| `GLOBAL_REMOTE` | `remote` AND (`anywhere` \| `global(ly)?` \| `worldwide` \| `work from anywhere`) or timezone-only constraints (`overlap with (ET\|PT\|EST\|PST\|UTC)`) with no country restriction |
| `SPONSORSHIP_POSITIVE` | affirms sponsorship and is NOT negated: `(will\|can\|do) sponsor`, `visa sponsorship (is )?available`, `H-?1B` in an affirmative clause. **Negation handling is mandatory** — "we do not sponsor H-1B" must never land here. |
| `US_REMOTE_ONLY` | `remote` AND (`US\|USA\|United States\|U\.S\.` in location, or authorization-required language) |
| `UNKNOWN` | none of the above — this is where the AI layer earns its keep |

Implementation notes:
- Regex pass first (free, deterministic). Only `UNKNOWN` and conflicting-signal cases go
  to the AI layer, and only when the API key is present.
- To keep AI tokens tiny, don't send whole descriptions: extract only lines containing
  `remote|location|visa|sponsor|authoriz|eligib|timezone|time zone|based in|work from`.
- Alert policy comes from `policy.yaml` (defaults above): instant Telegram for
  INDIA_ELIGIBLE / GLOBAL_REMOTE / SPONSORSHIP_POSITIVE; weekly digest for
  US_REMOTE_ONLY + UNKNOWN tier-1/2 titles (worth a manual look or a networking ping);
  drop onsite-no-sponsorship.

## AI layer (optional, `src/ai_score.py`)

- Runs only when `ANTHROPIC_API_KEY` is set; the pipeline must be fully functional
  without it (pure regex mode).
- Model: current Haiku (`claude-haiku-4-5` family — confirm the latest model string in
  Anthropic docs at build time). `temperature 0`, `max_tokens 300`.
- One call per job that (a) passed title filter and (b) is eligibility `UNKNOWN`, or
  (c) needs a fit score. Input: `profile.md` + title/company/location + extracted
  eligibility lines + first ~1500 chars of description.
- Output — demand strict JSON, no prose:
  ```json
  {"relevance": 0-100, "eligibility": "<enum value>", "seniority_fit": "junior|mid|senior|unclear", "one_liner": "why this does/doesn't fit"}
  ```
  Strip markdown fences before `json.loads`; on parse failure, retry once, then fall
  back to regex verdict + `score: null`.
- Volume estimate: after title filtering, expect roughly 5–40 calls/day at ~1–2K tokens
  each — trivial money at Haiku pricing, but verify current pricing at build time and
  print a per-run token count in the logs.
- Note for Amit: a Claude Code subscription does **not** provide API credits for
  scripts. This needs a separate API key from console.anthropic.com with a few dollars
  of prepaid credit — or just run in regex-only mode, which is genuinely fine.

## State & diffing (`src/state.py`)

- `state/seen.json`: `{job_id: {"first_seen": iso, "title": str, "company": str}}`.
- New job = id absent from state. Only new jobs are alerted/digested.
- Prune entries not seen in any fetch for 120 days (keeps the file small; jobs that
  reappear after that are treated as new, which is acceptable).
- **First-run seed mode:** if `seen.json` is empty, do NOT fire individual alerts.
  Seed the baseline and send ONE Telegram summary: total postings, companies fetched,
  endpoints failed, and a single digest listing the *current* tier-1/2 matches with
  alert-grade eligibility — immediate day-one value without 300 pings.
- The Action commits `state/seen.json` back to the repo after each run (bot identity,
  `[skip ci]` in the message, `concurrency` group prevents overlap).

## Notifications (`src/notify.py`)

Raw HTTPS, no library: `POST https://api.telegram.org/bot{TOKEN}/sendMessage` with
`chat_id`, `text`, `parse_mode=HTML`, `disable_web_page_preview=true`.

Per-match alert format:

```
🎯 <b>Clay</b> — GTM Engineer
📍 Remote (Global) · GLOBAL_REMOTE · Tier 1
⭐ 82 — Clay-native workflow role, junior-friendly, IST overlap OK
🔗 https://jobs.ashbyhq.com/clay/...
```

- If new matches > `max_alerts_per_run`, send top-N individually + one digest message
  for the rest.
- **Health alert:** if the run crashes or >20% of resolved endpoints fail, send
  `⚠️ job-radar unhealthy: <details>`. (GitHub also emails on workflow failure.)
- **Weekly digest (Sundays, same run):** US_REMOTE_ONLY/UNKNOWN tier-1/2 roles seen this
  week, tier-3 roles if enabled, endpoints currently failing, and counts.

```yaml
      - uses: actions/checkout@v4

          TELEGRAM_CHAT_ID: ${{ secrets.TELEGRAM_CHAT_ID }}

      - name: Commit state

          git config user.name "job-radar-bot"

          git add state/seen.json
          git diff --cached --quiet || git commit -m "radar: state update [skip ci]"
          git push
```

Known quirk: GitHub cron can fire 5–30 minutes late under load. Irrelevant for daily

## Resolver (`src/resolve.py`) — Phase 1, run once

Input: company names extracted from `data/GTM_Engineer_Companies_2026.md`.

1. **Extract & dedupe.** The doc repeats companies across sections (Rippling, Samsara,
   Veeva, Notion, Figma, Pendo, Brex appear twice) — dedupe by normalized name. Mark
   §9 agencies and the §4 mega-caps (Google Cloud, Microsoft, AWS, Oracle, SAP, IBM,
   Salesforce, Adobe, Workday, ServiceNow, NVIDIA) as `skip` by default — mega-caps are
   better covered by LinkedIn/HiringCafe alerts, agencies rarely run an ATS. Keep them
   in the YAML with `status: skip` so flipping one on later is a one-line edit.
2. **Slug candidates** per company: lowercase; strip spaces/punctuation/`.io`/`.ai`;
   hyphen and no-hyphen variants (e.g. `dbt Labs` → `dbtlabs`, `dbt-labs`; `Apollo.io`
   → `apollo`, `apolloio`; `6sense` → `6sense`).
3. **Probe** each candidate against each ATS pattern (Greenhouse → Lever → Ashby →
   Workable → SmartRecruiters → Recruitee). Valid = HTTP 200 + parses as JSON + has the
   expected top-level shape + returns ≥1 posting OR a plausible empty list. Beware
   false positives: some ATSes return 200 with an error body — check shape, not status.
4. **Fallback for misses:** fetch `https://{companydomain}/careers` (and `/jobs`,
   `/about/careers`) HTML with plain requests and grep for ATS URLs
   (`boards.greenhouse.io/X`, `jobs.lever.co/X`, `jobs.ashbyhq.com/X`,
   `apply.workable.com/X`, `myworkdayjobs.com` → mark `workday`). No browser — if the
   page is JS-only and nothing greps, mark `unresolved`.
5. **Output** `config/companies.yaml` + print a resolution report:
   resolved-by-ATS counts, `workday` list, `unresolved` list. Expect roughly 70–80% of
   the non-skipped list to resolve on the first pass; unresolved ones get handled
   manually or asked about in a Claude chat session.

## Test cases (`tests/test_filters.py`) — must pass

Title filter:
- "GTM Engineer" → tier 1 · "Go-To-Market Engineer" → tier 1 · "Growth Engineer" → tier 1
- "Senior GTM Engineer" → tier 1 + `senior_downrank` flag (not excluded)
- "Director, Revenue Operations" → hard_exclude
- "GTM Operations Manager" → tier 2 (manager alone is NOT excluded)
- "Technical Growth Lead" → tier 2 (tier match beats the `lead` downrank)
- "Solutions Engineer" → tier 3 · "Software Engineer, Growth" → tier 1 (order-insensitive match)
- "GTM Engineer Intern" → tier 1

Eligibility:
- "Remote (US)" → US_REMOTE_ONLY · "Remote — United States" → US_REMOTE_ONLY
- "Remote - Anywhere" → GLOBAL_REMOTE · "Fully remote, work from anywhere" → GLOBAL_REMOTE
- "Bengaluru, India" → INDIA_ELIGIBLE · "Remote (APAC)" → INDIA_ELIGIBLE
- "New York, NY" + "visa sponsorship available for exceptional candidates" → SPONSORSHIP_POSITIVE
- "San Francisco" + "we are unable to sponsor visas at this time" → SPONSORSHIP_NEGATIVE_ONSITE
- "Remote" + "must be authorized to work in the United States without sponsorship" → US_REMOTE_ONLY (not sponsorship_positive — negation handled)
- "" (empty location) → UNKNOWN
